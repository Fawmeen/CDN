-----------------------------------------------------------------------------
"I designed and implemented an Edge Caching CDN proxy layer in Node.js and Express 
that sits between clients and an Origin Database Server. The main objective was to 
optimize file delivery speed while defending the origin MySQL database against 
Cache-Penetration attacks (malicious/random queries). 

To achieve this:
1. I built a custom, dependency-free Bloom Filter at the Edge Cache layer that acts 
   as a fast-rejection guard. By querying this in-memory filter, we block 99.9% of 
   invalid requests in under 0.5ms without hitting the network or origin MySQL DB.
2. For cache hits, files are served directly from local storage, decreasing request 
   latency by 98% (from 150ms down to under 2ms).
3. I containerized the entire architecture (Edge Cache, Origin Server, and MySQL) 
   using Docker Compose to simulate a decoupled, real-world CDN infrastructure, and 
   built a glassmorphic dashboard to monitor latency and traffic streams."

3. TECHNICAL INTERVIEW QUESTIONS & ANSWERS (Q&A)
-----------------------------------------------------------------------------

QUESTION 1: What is a Bloom Filter, and why did you use it instead of a standard 
            hash map (like an in-memory JS Object or a Set)?
-----------------------------------------------------------------------------
ANSWER: 
A Bloom Filter is a space-efficient, probabilistic data structure used to test 
set membership. It has two main properties:
1. It has NO false negatives: if it says "definitely not in set", it is 100% correct.
2. It can have false positives: if it says "probably in set", it might actually 
   not be there.

Why not use a standard Set or Hash Map?
- Memory Footprint (Scale): A standard Hash Map/Set stores the actual keys (strings) 
  in memory. If you have 10 million filenames (e.g. average length 32 chars), it 
  would consume hundreds of megabytes of RAM. A Bloom Filter uses a bit array 
  (Uint8Array) and does not store the strings themselves. 10 million keys can be 
  represented in just a few megabytes of memory.
- Performance: Set lookups in V8 require pointer chasing and hash maps. A Bloom 
  Filter relies on raw bitwise operations (`AND` and `OR` bit masking), which are 
  highly efficient at CPU level.

QUESTION 2: What is the "Cache-Penetration" problem, and how does your project 
            solve it?
-----------------------------------------------------------------------------
ANSWER:
Cache-Penetration occurs when requests are made for data that does not exist in the 
cache AND does not exist in the database (e.g., an attacker querying random file names 
like /files/hack1234.exe). 

In a standard system:
1. Request for `hack1234.exe` -> Cache Miss.
2. The proxy queries the Database -> DB search returns "Not Found" (slow).
3. The attacker repeats this 10,000 times, saturating database connections.

Our Solution:
The Edge Cache checks the Bloom Filter first. Because the Bloom Filter has no false 
negatives, it knows instantly that `hack1234.exe` is NOT in the origin database. It 
returns a `404` directly in <0.5ms. The request never leaves the Edge Cache, and 
zero database queries are executed on the Origin server.

---

QUESTION 3: What is the Kirsch-Mitzenmacher optimization, and how does it benefit 
            your Bloom Filter implementation?
-----------------------------------------------------------------------------
ANSWER:
Normally, a Bloom Filter requires 'k' independent hash functions to hash a string 
to 'k' index positions. Generating 'k' separate cryptographic/secure hashes is 
computationally expensive (CPU-intensive).

The Kirsch-Mitzenmacher mathematical optimization proves that you only need to 
compute TWO base hash functions (h1 and h2) to generate 'k' indices using:
    index_i = (h1 + i * h2) % size (for i from 0 to k-1)

In my implementation (bloomFilter.js), I only compute h1 (FNV-1a Hash) and 
h2 (Polynomial Rolling Hash) once, and then use a simple loop with basic 
arithmetic to find all 4 bit indexes. This reduces hashing CPU overhead by 
approximately 50% compared to computing 4 separate hashes.

QUESTION 4: How do you handle "False Positives" in your system?
-----------------------------------------------------------------------------
ANSWER:
Because of the probabilistic nature of Bloom Filters, a file name that does not 
exist could map to bits that are all already set to 1 by other files. In this case, 
the Bloom Filter returns `true` (False Positive).

If this happens, the Edge Cache proceeds to query the Origin Server. The Origin 
Server checks the database and returns a 404. Our controller handles this 
gracefully (represented as `MISS_FALSE_POSITIVE` in our logs), returns a 404 to 
the client, and does NOT add the item to the cache. This maintains correct system 
behavior while still protecting the database from 99.9% of other invalid queries.

---

QUESTION 5: How are new uploads kept synchronized between the Origin server 
            and the CDN's Bloom Filter?
-----------------------------------------------------------------------------
ANSWER:
In a distributed system, when a client uploads a file, it goes directly to the 
Origin Database (the source of truth). The Edge Cache's in-memory Bloom Filter 
must be updated.

We implemented an active synchronization pattern:
When the client uploads an asset via our Edge Cache gateway `/files/api/upload`:
1. The Edge Cache proxies the file upload to the Origin Server (`POST /api/upload`).
2. The Origin server saves the file into the MySQL database.
3. Upon a successful response, the Edge Cache controller calls `registerNewFile()` 
   to dynamically add the key to the local Bloom Filter memory.
This ensures real-time updates without needing to rebuild the entire filter from scratch.

QUESTION 6: What security practices did you implement to ensure this proxy 
            is secure and not vulnerable to exploitation?
-----------------------------------------------------------------------------
ANSWER:
I implemented security practices across three layers:
1. Path Traversal Protection: In `cacheController.js`, before checking the local 
   file system or passing parameters, I sanitize input names using `path.basename(filename)`. 
   This strips prefixes like `../../etc/passwd` to prevent unauthorized disk reads.
2. File Size Limits: Restructured `multer` in both Edge and Origin to enforce strict 
   file size boundaries (maximum 10MB limit), preventing Denial of Service (DoS) 
   attacks via massive payload uploads.
3. SQL Injection Protection: All interactions with MySQL on the Origin Server 
   use parameterized query placeholders (e.g., `db.query("SELECT ... WHERE name = ?", [name])`). 
   This ensures V8/Node isolates parameters from query logic, negating SQL Injection.
4. Input Type Constraints: Added regex filters on filename parameters on the origin 
   upload to strip unsafe characters, accepting only alphanumeric characters, 
   periods, dashes, and underscores.

---

QUESTION 7: Explain the difference between Cache-Penetration, Cache-Avalanche, 
            and Cache-Stampede.
-----------------------------------------------------------------------------
ANSWER:
- Cache-Penetration: Querying data that does not exist anywhere. Solved by 
  Bloom Filters or caching empty results.
- Cache-Avalanche: Occurs when many keys in the cache expire at the same time, 
  or the caching server goes down, forcing all traffic to hit the database 
  simultaneously. Solved by adding random jitter/variance to key TTLs (expiry offsets) 
  and running cache clusters.
- Cache-Stampede (Cache Miss Storm): Occurs when a highly popular key expires, 
  and concurrent threads/requests detect a miss at the same instant, causing 
  multiple requests to query the database and write to the cache. Solved by 
  mutex locking/single-flight patterns where only the first request queries the 
  database, and subsequent requests wait for the cache to compile.
