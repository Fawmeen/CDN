2. DETAILED PERFORMANCE ANALYSIS
------------------------------------------------------------------------

I. Bloom Filter Efficiency (The Guard Layer)
--------------------------------------------
- Average Server Latency: 0.86 ms
- Worst-Case Server Latency (Max): 1.56 ms (under 100 concurrent requests)
- Performance Speedup: ~35.3x faster than a Cache Miss

Analysis:
The Bloom Filter functions as an anti-cache-penetration guard at the Edge Cache.
When requests for non-existent files are received, the Bloom Filter instantly blocks
them in under 1 ms, preventing a network round-trip request to the Origin server
and avoiding a database check. By serving rejections in 0.86 ms instead of the 30.37 ms
required for a Cache Miss database lookup, the system processes requests 3,500%
faster. This prevents database resource exhaustion under high request volume or
malicious search attacks.


II. Caching Performance (Disk HIT vs. DB MISS)
----------------------------------------------
- Cache HIT (Disk): 12.80 ms
- Cache MISS (Database Lookup at Origin): 30.37 ms
- Performance Speedup: ~2.37x faster than database queries

Analysis:
Serving cached assets directly from the local disk of the CDN Edge node is more than
twice as fast as fetching them from the origin server's MySQL database.
However, under a concurrent load of 100 requests, a minor Disk I/O bottleneck
becomes visible: while the minimum cache hit time is 7.75 ms, the worst-case (Max)
spikes to 48.49 ms (p99 is 48.47 ms). Node's filesystem reading threads are blocked
waiting on disk operations.
To scale this caching proxy to thousands of concurrent requests, replacing local disk
caching with an in-memory Redis caching store is recommended to achieve sub-millisecond
hit latencies under high concurrency.


III. Network & WSL2 Virtualization Overhead
-------------------------------------------
Analysis:
There is a significant difference between Client-side roundtrip latency (971.51 ms Avg)
and Server-side internal processing latency (12.80 ms Avg) for Cache Hits.
This overhead is caused by:
1. Docker WSL2 VM Network Bridge: On Windows, Docker containers run inside a WSL2
   Linux virtual machine. Requests sent from the Windows host must cross the host-to-VM
   boundary and be routed through Docker's internal networking layer, creating routing
   queues under concurrent bursts.
2. Connection Pooling: Node's native fetch handler queues concurrent connections to a
   single host, adding client-side delay.
The Server-Side Internal Processing Latency isolates this network bridge lag, proving
that the backend server code executes cache hits in 12.80 ms and bloom rejects in 0.86 ms.
